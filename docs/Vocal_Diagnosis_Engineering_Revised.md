## 第一章：ChatGPT 反馈的五个核心问题及回应

### 问题 1："很多声学特征在真实环境中不稳定"

| ChatGPT 原文 | 我们的回应 |
|-------------|-----------|
| "HNR < 12 → 挤卡" 在真实环境里会炸 | **完全同意**。HNR 是弱特征，不是决定性特征。V0.1 用它只因为我们没有数据，一旦有 DL 模型立刻替换。 |
| 麦克风质量、混响、背景噪音都会影响 | **已纳入**。第五章专门讨论环境鲁棒性方案。 |
| "你没挤卡，但它一直红灯" 是产品死亡级 bug | **完全同意**。我们的对策：宁可漏报不可误报（precision > recall）。 |

**修改措施**：
- V0.1 的规则引擎加入**环境噪声估计**和**自动阈值校准**
- HNR 阈值从固定 12dB 改为**动态阈值** = 用户前 30 秒的 HNR 中位数 - 偏移量
- 增加**置信度消退**：如果环境噪声估计 > -30dB，自动降低检测灵敏度

### 问题 2："高估了单帧声学特征的判别能力"

| ChatGPT 原文 | 我们的回应 |
|-------------|-----------|
| 声乐问题 = 时序行为，不是单帧决策 | **完全同意**。挤卡是持续+累积的模式，不是一帧 HNR 低就算。 |
| 过度"帧级诊断"，不足"时序建模" | **已有部分方案**（EMA + 状态机），但需要进一步加强。 |

**修改措施**：
- V0.1：加强 EMA + 状态机（已有，但参数需要基于真实数据调优）
- V0.2：CNN 输入本身就是 32 帧 = 512ms 窗口（天然时序建模）
- V1.0：考虑引入 phrase-level 特征（乐句级诊断，不只是帧级）
- 第七章详细讨论时序建模方案

### 问题 3："部分指标工程上不可稳定获取"

| 指标 | ChatGPT 评价 | 我们的决策 |
|------|-------------|-----------|
| CPP | 端侧不稳定 | ✅ 已在 V0.1 中**弃用**（只保留 debug） |
| F1/F2/F3 (共振峰) | 高音时不稳定 | ⚠️ 降级为 **DL 隐式特征**，不做显式提取 |

- 可靠特征（V0.1 可用）：**只有 Mel、HNR、Energy、F0**
- 准可靠特征（V0.3 引入）：**H1-H2**（但需要 F0 精确）
- 不可靠特征（永远只在 DL 里隐式学习）：**CPP、共振峰、Jitter/Shimmer**

## 第二章：新的诊断哲学 — 从"特征阈值"到"模式识别"

### 2.1 旧哲学（危险的）

```
旧思路：
  HNR < 12 → 挤卡
  H1-H2 > 0 → 漏气
  F1 shift > 15% → 喉位高

问题：
  ① 同一个问题在不同人/环境下特征值差异极大
  ② 阈值固定 → 有人永远红灯，有人永远绿灯
  ③ 单一特征 → 混淆（HNR 低可以是挤卡、漏气、噪声环境）
```

### 2.2 新哲学

```
新思路（渐进式）：

V0.1（别无选择，只能用规则）：
  HNR + Energy + F0 → 规则引擎
  但是！加上以下防御措施：
    ① 动态阈值（基于用户短期基线）
    ② 多特征交叉验证（HNR 低 + Energy 高 → 更可能是挤卡）
    ③ 高置信区间才报警（宁可不说，不可说错）
    ④ 环境噪声估计（如果太吵就不下判断）

V0.2（关键转折点）：
  CNN 从 Mel 频谱中学习"模式"
  不再依赖单一特征的阈值
  而是学习"挤卡的声音长什么样"的整体模式
  
  好处：
    ① CNN 自动适应不同麦克风特性
    ② CNN 看到的是完整的频谱形状，不是 1 个数字
    ③ 32 帧窗口提供了时序上下文

V1.0：
  多任务 CNN → 同时学习多个维度的模式
  维度之间共享特征 → 互相增强
```

### 2.3 核心公式变化

```
旧公式：
  problem = threshold(single_feature)
  
新公式：
  V0.1: problem = state_machine(ema(cross_validated_rule(features)))
  V0.2: problem = state_machine(ema(cnn(mel_context)))
```

## 第三章：可稳定检测的维度（收敛到 5 个）

以下 5 个维度是经过"工程可行性拷问"后保留的核心维度。

### 维度 1：紧张度（原"挤卡"）

| 项目 | 修改前 | 修改后 |
|------|--------|--------|
| 名称 | 挤卡检测 | **紧张度评分** |
| 输出 | 二值（红/绿） | **0-100 分连续评分** |
| 算法 | HNR < 12 → 挤卡 | **多特征交叉 + 动态阈值 + 高置信区间** |
| 用户表达 | "你在挤卡" | **"紧张度 72/100"** |

**为什么改名？**

```
"挤卡" 是一个诊断性判断 → 用户会质疑"你怎么知道我在挤卡？"
"紧张度" 是一个描述性指标 → 用户更容易接受"我的声音听起来紧张"

而且：
  紧张度 = 更宽的概念，cover 了挤卡、假声带参与、声门冲击等
  用户不需要知道具体是哪种 → 只需要知道"放松一点"
```

**V0.1 修改后的算法**：

```python
def compute_tension_score(hnr, energy, f0, env_noise_estimate, user_baseline_hnr):
    """
    修改后的紧张度计算：
    ① 加入环境噪声估计
    ② 加入用户基线自适应
    ③ 多特征交叉验证
    ④ 输出 0-100 分而非概率
    """
    # Gate: 静音和无声段不判断
    if energy < -40 or f0 < 50:
        return None  # 不输出分数（不是 0 分！而是"无法判断"）
    
    # 环境噪声防护
    if env_noise_estimate > -25:
        return None  # 环境太吵，不下判断
    
    # 动态阈值：基于用户前 30 秒的 HNR 中位数
    dynamic_threshold = max(user_baseline_hnr - 6, 8.0)  # 不低于 8
    
    # 核心概率
    raw_prob = sigmoid(0.5 * (dynamic_threshold - hnr))
    
    # 交叉验证：挤卡通常伴随高能量
    # 如果 HNR 低但能量也低 → 更可能是漏气/噪声，不是挤卡
    if hnr < dynamic_threshold and energy < -20:
        raw_prob *= 0.5  # 降低置信度
    
    # 高音补偿
    if f0 > 220:
        raw_prob *= 1.0 + 0.3 * min((f0 - 220) / 300, 1.0)
        raw_prob = min(raw_prob, 1.0)
    
    # 输出 0-100 分（100 = 完全放松，0 = 极度紧张）
    tension_score = round(100 * (1 - raw_prob))
    return tension_score
```

**关键改进点**：

| 改进 | 作用 | 解决原文档的哪个问题 |
|------|------|-------------------|
| 动态阈值 | 适应不同用户嗓音 | 问题 1（特征不稳定） |
| 环境噪声估计 | 太吵时不判断 | 问题 1（环境干扰） |
| 交叉验证 | 减少单特征误判 | 问题 2（过度依赖 HNR） |
| 改为评分不判诊断 | 用户更容易接受 | 产品风险（用户不信你） |
| None 返回（不判断） | 宁缺毋滥 | 产品风险（红灯乱闪） |

### 维度 3：稳定性

| 项目 | 修改前 | 修改后 |
|------|--------|--------|
| 名称 | 气息稳定性 + 混声稳定性（分开的两个） | **合并为"声音稳定性"** |
| 算法 | 能量 CV + F0 CV | **多信号 CV + 排除颤音** |
| 用户表达 | "气息不稳" | **"稳定性 85/100"** |

**修改理由**：
- "气息不稳"是诊断性判断，用户会质疑
- "稳定性"是客观可衡量的描述
- 多个"不稳定"的原因可以合并（用户只需知道"不稳定"，不需要知道是气息还是声带）

**算法**：

```python
def compute_stability_score(f0_array, energy_array, frame_rate=62.5):
    """
    在 0.5 秒窗口内计算稳定性
    """
    window_frames = int(0.5 * frame_rate)  # ~31 frames
    
    # F0 稳定性（排除正常颤音 4.5-6.5Hz）
    f0_detrended = remove_vibrato(f0_array, frame_rate)  # 去除正常颤音分量
    f0_cv = np.std(f0_detrended) / (np.mean(f0_array) + 1e-8)
    
    # Energy 稳定性
    energy_cv = np.std(energy_array) / (np.mean(energy_array) + 1e-8)
    
    # 综合稳定性（加权）
    instability = 0.6 * f0_cv + 0.4 * energy_cv
    stability_score = max(0, min(100, round(100 * (1 - instability * 10))))
    
    return stability_score
```

```
  让 CNN 从 Mel 频谱中隐式学习"闭合不全的声音模式"
  不再依赖 H1-H2 这个不稳定的手工特征
```

### 维度 5：能量控制

| 项目 | 修改前 | 修改后 |
|------|--------|--------|
| 名称 | 力度控制 + 大白嗓（分开） | **合并为"能量控制评分"** |
| 算法 | Energy jitter + H1-H2 + Singer's Formant | **Energy + 频谱形状（简化）** |
| 用户表达 | "力度失调" / "你在喊" | **"能量控制 60/100" + "试着控制音量"** |

**合并理由**：
- "力度控制差" 和 "大白嗓" 在用户体验上的表现很接近："太用力了"
- 分开检测需要 H1-H2 和 Singer's Formant → 不够稳定
- 合并后只需要 Energy + Mel 频谱 → 更鲁棒

**V0.2 算法**：

```python
def compute_energy_control_score(energy_array, mel_array):
    """
    能量控制 = 音量的合理性 + 稳定性
    """
    # 1. 能量过高检测（大白嗓倾向）
    peak_energy = np.max(energy_array)
    mean_energy = np.mean(energy_array)
    energy_headroom = peak_energy - mean_energy
    
    # 如果峰值远高于平均 → 有冲击性的"喊"
    impulse_penalty = max(0, (energy_headroom - 6) / 12)  # 超过 6dB 就扣分
    
    # 2. 频谱平衡（简化版 Singer's Formant）
    # 不用精确的 3kHz → 用 Mel 频谱的高频/低频比
    high_band_energy = np.mean(mel_array[40:60])  # Mel band 40-60 ≈ 1.5-4kHz
    low_band_energy = np.mean(mel_array[0:20])     # Mel band 0-20 ≈ 0-500Hz
    spectral_balance = high_band_energy - low_band_energy  # dB
    
    # 完全靠低频 → 可能是大白嗓
    # 高频有聚焦 → 有共鸣
    
    # 3. 能量 jitter
    energy_diffs = np.abs(np.diff(energy_array))
    energy_jitter = np.std(energy_diffs)
    jitter_penalty = max(0, (energy_jitter - 2) / 6)
    
    # 综合评分
    raw_score = 1.0 - 0.4 * impulse_penalty - 0.3 * jitter_penalty
    # spectral_balance 的影响在 V0.2 用 DL 隐式学习
    
    score = max(0, min(100, round(100 * raw_score)))
    return score
```

## 第四章：不可稳定检测的维度（诚实降级）

| 维度 | 原计划版本 | 修改后版本 | 降级原因 |
|------|-----------|-----------|---------|
| **喉位高/低** | V1.0 | **V2.0** | 共振峰在唱歌中极不稳定。Mel CNN 可能隐式学到，但显式检测不可靠 |

| **假声带振动** | V1.0 | **V2.0** | 次谐波检测需要训练数据，且极少见 |

## 第五章：环境鲁棒性工程方案

这是 ChatGPT 反馈中最尖锐的问题："同一段音频在不同环境下结果不同 → 产品死亡"

### 5.1 环境噪声估计

```python
class EnvironmentNoiseEstimator:
    """
    实时估计环境底噪
    方法：追踪最近 5 秒内能量最低的 10% 帧的平均能量
    """
    def __init__(self, history_seconds=5, percentile=10):
        self.buffer = []  # 存储最近 N 帧的能量
        self.max_frames = int(history_seconds * 62.5)
    
    def update(self, frame_energy_db):
        self.buffer.append(frame_energy_db)
        if len(self.buffer) > self.max_frames:
            self.buffer.pop(0)
    
    def estimate(self):
        if len(self.buffer) < 30:
            return -50  # 数据不够，假设安静
        sorted_energies = sorted(self.buffer)
        low_10pct = sorted_energies[:max(3, len(sorted_energies) // 10)]
        return sum(low_10pct) / len(low_10pct)
```

### 5.2 环境分级策略

| 环境噪声估计 | 分级 | App 行为 |
|-------------|------|---------|
| < -45 dB | 🟢 安静 | 全功能检测 |
| -45 ~ -35 dB | 🟡 一般 | 正常检测，HNR 阈值放宽 2dB |
| -35 ~ -25 dB | 🟠 较吵 | 只输出 Energy 相关指标，紧张度不显示 |
| > -25 dB | 🔴 太吵 | 提示用户"环境太吵，建议换个安静的地方" |

### 5.3 用户基线自适应

```
原来：
  HNR 阈值 = 固定 12dB（所有人相同）

修改后：
  App 启动后前 30 秒为"校准期"
  收集用户在"舒适区"（中音、正常音量）的 HNR 分布
  user_baseline_hnr = median(hnr_values_in_comfort_zone)
  
  动态阈值 = max(user_baseline_hnr - 6, 8.0)
  
  效果：
  - 嗓音条件好的人（baseline HNR = 20）→ 阈值 = 14dB
  - 嗓音条件差的人（baseline HNR = 14）→ 阈值 = 8dB
  - 避免了"嗓音条件差的人永远红灯"的问题
```

### 5.4 麦克风频响补偿

```
不同手机麦克风的频率响应不同：
  iPhone 15 Pro: 较平坦
  iPhone 12 mini: 高频偏弱
  在耳机麦克风: 低频偏弱

对 HNR/Mel 的影响：
  高频弱 → HNR 偏低 → 误判挤卡
  低频弱 → H1 偏低 → H1-H2 不准

解决方案（V0.2+）：
  Step 1: 在校准期，让用户发一个 /a/ 音
  Step 2: 计算频谱包络
  Step 3: 与标准参考频谱对比
  Step 4: 估计频率响应偏差
  Step 5: 对后续特征做补偿

  注意：V0.1 不做（太复杂），先用动态阈值兜底
```

## 第六章：从"诊断"到"评分" — 产品表达重构

ChatGPT 最有价值的建议之一：**不要说"你有问题"，而是说"你的当前状态"**。

### 6.1 语言重构

| ❌ 旧表达（诊断式） | ✅ 新表达（评分式） |
|-------------------|-------------------|
| "你在挤卡" | "紧张度 72/100" |
| "你闭合不严" | "声音偏虚——可以试着多一些支撑" |
| "你喉位太高" | "共鸣空间偏小——打个哈欠的感觉试试" |
| "你在大白嗓" | "能量控制 45/100——试着用共鸣代替力量" |
| "你气息不稳" | "稳定性 68/100——长音再稳一点" |

### 6.2 评分仪表盘设计

```
╔══════════════════════════════════╗
║  🎤 实时状态                      ║
║                                  ║
║  紧张度    ████████░░  78/100     ║
║  稳定性    ██████░░░░  60/100     ║
║  能量控制  █████████░  85/100     ║
║                                  ║
║  综合评分  ██████████  74/100     ║
║                                  ║
║  💡 "放松一点，声音会更自然"        ║
╚══════════════════════════════════╝
```

### 6.3 反馈文案分级

```
90-100: "🟢 非常棒，保持！"
70-89:  "🟡 还不错，再放松一点"
50-69:  "🟠 有一些紧张——深呼吸，让喉咙打开"
30-49:  "🔴 比较紧张——试试用哼鸣放松一下"
0-29:   "🔴 需要休息——喝口水，重新来"
```

### 6.4 关键原则

| 原则 | 解释 |
|------|------|
| **永远正向框架** | 不说"你做错了"，说"试试这样" |
| **用分数不用诊断** | 72/100 比 "挤卡" 更容易接受 |
| **不确定就不说** | 环境吵/帧太少/置信度低 → 显示 "—" |
| **进步是最好的激励** | "今天比昨天好了 5 分" 比 "你还是在挤卡" 好 1000 倍 |

## 第七章：时序建模 — 修复帧级诊断的致命缺陷

ChatGPT 指出的核心问题：声乐问题是**时序行为**，不是帧级事件。

### 7.1 时序层次

```
                ms 级（帧级）
                    │
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
  帧内特征        帧间平滑        事件检测
  (HNR/Mel/F0)   (EMA/State)    (破音/起音)
                    │
                    ↓
              0.5-2 秒级（短时行为）
                    │
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
  稳定性CV       颤音FFT        CNN窗口
  (0.5s)         (1s)           (512ms)
                    │
                    ↓
              5-30 秒级（乐句级）
                    │
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
  能量衰减趋势    换声行为       气息分配
  (phrase)        (passaggio)    (phrase)
                    │
                    ↓
             分钟级（练习级）
                    │
                    ↓
              疲劳检测 / 进步检测
```

### 7.2 V0.1 的时序方案（已有，增强）

```
Level 1: 帧级
  特征计算（无时序）

Level 2: EMA 平滑 (当前)
  α = 0.4 → 2-3 帧的等效平滑
  优点：简单、低延迟
  缺点：只是"模糊化"，不是真正的时序建模

Level 3: 状态机 (当前)
  trigger: 6 帧 > 0.55
  clear: 10 帧 < 0.30
  优点：避免闪烁
  缺点：固定帧数，不适应不同速度的乐句

新增 Level 2.5: 滑动窗口统计
  在 0.5 秒窗口内计算 HNR 的中位数（比 EMA 更鲁棒于瞬态噪声）
  中位数对异常值不敏感 → 一帧噪声不会拉低整个窗口
```

### 7.3 V0.2 的时序方案（CNN 天然时序）

```
CNN 输入 [82 × 32]：
  32 帧 = 512ms
  
  这意味着 CNN 看到的是：
  "过去 512ms 的完整频谱变化过程"
  
  而不是：
  "这一帧 HNR = 8dB"
  
  CNN 可以学到：
  ① 挤卡通常是"HNR 逐渐下降"的模式（不是突然一帧低）
  ② 声区转换时的频谱"过渡模式"
  ③ 稳定音和不稳定音的区别
```

### 7.4 V1.0 的时序方案（乐句级）

```
新增：乐句边界检测
  通过 energy 静音段（< -40dB, > 150ms）切分乐句
  
乐句级特征：
  phrase_duration: 乐句时长
  phrase_energy_slope: 乐句内能量趋势（气息充足性）
  phrase_f0_range: 乐句内音高范围
  phrase_tension_trajectory: 紧张度随时间的变化
  
乐句级诊断（V1.0 新增）：
  if phrase_energy_slope < -8dB/s → "这句气息不太够"
  if phrase_tension_trajectory 从低到高 → "这句越唱越紧张（可能是音高上升导致）"
```

## 第八章：标注一致性问题及解决方案

ChatGPT 指出：声乐概念在专家之间都不一致，label noise 很高。

### 8.1 问题量化

| 维度 | 预期 Cohen's κ | 标注难度 |
|------|---------------|---------|
| 挤卡（明显的） | 0.7-0.9 | 低（大家都能听出来） |
| 挤卡（轻度的） | 0.4-0.6 | 中 |
| 混声比例（5级） | 0.3-0.5 | 高（争议大） |
| 漏气 | 0.6-0.8 | 中 |
| 喉位 | 0.3-0.5 | 高（需要经验） |
| 鼻音 | 0.4-0.6 | 中高 |

### 8.2 解决方案

```
方案 1: 只标注高一致性的维度
  κ < 0.6 的维度 → 暂不进入训练集
  先训练挤卡（κ 最高）→ 积累经验后再扩展

方案 2: 多标注者投票
  每个 clip 由 3 个标注者独立标注
  只保留至少 2/3 一致的 clip
  丢弃不一致的 → 数据少但标签干净

方案 3: 简化标注粒度
  混声比例：5 级 → 3 级（胸/混/头）
  挤卡：连续概率 → 二值（有/没有）
  粒度越粗 → κ 越高

方案 4: 标注者校准
  每 100 clip 插入 10 个"黄金标准" clip（已知正确答案）
  标注者在黄金标准上的准确率 < 80% → 暂停 + 重新培训

方案 5: Label Smoothing 训练
  标签不是 hard 0/1，而是 soft（0.1/0.9）
  允许模型有一定的"犹豫"余地
  loss = BCE(model_out, 0.9)  # 而不是 BCE(model_out, 1.0)
```

### 8.3 弱标签训练策略

```
数据层次：
  Tier 1 (Gold): 声乐教师标注，3 标注者一致 → 权重 1.0
  Tier 2 (Silver): 规则引擎标注 + 用户反馈一致 → 权重 0.5
  Tier 3 (Bronze): 规则引擎单独标注 → 权重 0.2

训练 loss = 
  Σ weight_i × BCE(model(x_i), label_i)
```

## 第九章：修订后的版本演进路线

### 原路线 vs 修改后路线

```
原路线（过于激进）：
  V0.1: 1 维度 (规则)
  V0.2: 6 维度 (规则+DL)
  V0.3: 7 维度
  V1.0: 19 维度
  
修改后路线（现实主义）：
  V0.1: 1 维度 (紧张度评分 — 规则+动态阈值+环境防护)
  V0.2: 3 维度 (紧张度 + 混声 + 稳定性)  ← CNN 上线
  V1.0: 5 维度 (+漏气 + 能量控制)         ← 多任务 CNN
  V1.5: 7 维度 (+颤音 + 破音)             ← 传统DSP可靠维度
  V2.0: 10 维度 (+喉位 + 鼻音 + 面罩)     ← 数据充足后
  V3.0: 15+ 维度 (+ASR辅助的维度)
```

### 每个版本的"上线门槛"

| 维度 | 上线门槛 | 测量方式 |
|------|---------|---------|
| 紧张度 | 在 100 个测试 clip 上，precision > 80% | 声乐教师标注作为 ground truth |
| 混声 | 5 分类 top-1 accuracy > 60% | 3 位教师投票标注 |
| 稳定性 | 与人耳判断一致率 > 75% | A/B 盲听测试 |
| 漏气 | precision > 75% 且与挤卡的混淆率 < 10% | — |
| 能量控制 | 与"用力过猛"标注一致率 > 70% | — |

**核心原则**：一个维度达不到门槛，宁可不上线。半准的功能比没有功能更糟糕。

## 第十章：自我检查清单 — "能上线吗？"

每个版本发布前，过一遍这个清单：

### 10.1 工程可行性检查

- [ ] 在 3 种不同 iPhone 型号上，同一段音频的检测结果偏差 < 10%？
- [ ] 在安静房间 vs 普通房间，结果偏差 < 15%？
- [ ] 环境太嘈杂时，App 是否正确拒绝检测（显示"—"而非错误结果）？
- [ ] 连续运行 30 分钟是否有内存泄漏或性能退化？
- [ ] 端到端延迟是否 < 100ms？

### 10.2 检测精度检查

- [ ] 在测试集上，每个上线维度的 precision > 门槛？
- [ ] 在"明显不是该问题"的负例上，误报率 < 5%？
- [ ] 在边界案例（轻度问题）上，宁可不报也不误报？
- [ ] 用户基线校准是否有效？（同一用户多次校准结果一致？）

### 10.3 产品体验检查

- [ ] 红灯是否有闪烁问题？（同一音保持时，灯不应快速切换）
- [ ] 分数变化是否平滑？（不应帧间跳动超过 10 分）
- [ ] "无法判断"的场景是否正确显示"—"而非错误数值？
- [ ] 文案是否全部是正向框架？（不说"你做错了"）
- [ ] 练习建议是否具体可行？（不说"放松"而是"试试唇颤音"）

### 10.4 极端场景检查

- [ ] 用户在地铁上打开 → 提示"环境太吵"而非乱报？
- [ ] 用户完全不发声 → 不出分数（不是 100 分）？
- [ ] 用户说话而非唱歌 → 不出分数或给出提示？
- [ ] 用户嗓音沙哑（感冒）→ 不会永远红灯？
- [ ] 用户手捂麦克风 → 不会误判为挤卡？

## 附录：与原文档的差异总结

| 维度 | 原文档 | 本文档 | 改变原因 |
|------|--------|--------|---------|
| 哲学 | 特征阈值驱动 | **模式识别 + 动态阈值** | 单一阈值在真实环境不稳定 |
| 检测数量 (V1.0) | 19 个 | **5 个** | 数据不够、标注不一致 |
| 输出形式 | 二值诊断 | **0-100 分评分** | 用户更容易接受 |
| 环境鲁棒 | 未提及 | **多层防护** | 不同手机/环境结果不一致是产品死亡 |
| 时序建模 | 帧级为主 | **多层次时序** | 声乐问题是时序行为 |
| 标注 | 假设可直接标注 | **多方案应对 label noise** | 专家之间一致性低 |
| CPP | debug only | **永久弃用** | 不可靠 |
| 共振峰 | 显式提取 | **仅 DL 隐式** | 唱歌中太不稳定 |
| H1-H2 | V0.3 规则 | **V1.0 DL 隐式** | 手工特征不够稳定 |
