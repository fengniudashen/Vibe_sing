## 1. 产品概述

### 1.1 一句话定义

Vibesing（高音觉醒）是一款 iOS 实时声乐诊断 App，通过端侧 AI 声学分析，帮助用户发现并消除高音时的发声障碍，逐步"觉醒"自己的高音能力。

### 1.2 产品定位

| 维度 | 定位 |
|------|------|
| **品类** | 声乐训练工具（非 K 歌娱乐） |
| **核心差异** | 不看音准，看发声机能——全球首个实时检测"挤卡"等发声病灶的消费级 App |
| **平台** | iOS 优先（iPhone + iPad），后续扩展 Android |
| **网络依赖** | 核心功能全部离线运行（端侧推理） |
| **语言** | 中文优先，英文同步支持 |

### 1.3 产品 Slogan

> **"每个人都有高音，只是没有觉醒。"**

## 2. 市场背景与机会

### 2.1 市场规模

- 全球在线音乐教育市场：2025 年约 $50 亿，年增长率 18%+
- 中国 K 歌/声乐 App 日活用户：~2 亿（全民 K 歌 + 唱吧等）
- 声乐培训线下市场（中国）：~300 亿 RMB/年
- 目标切入点：从"想唱好但不知道哪里有问题"的大众用户开始

### 2.2 痛点分析

| 用户痛点 | 现有解决方案 | 问题 |
|---------|------------|------|
| "我高音唱不上去" | 声乐老师 | 贵（¥300+/节），排课难 |
| "我高音唱不上去" | K 歌 App 打分 | 只看音准，不看发声方式 |
| "我知道自己在喊，但不知道怎么改" | YouTube/B 站教学 | 无个性化反馈，看了也不知道自己做对没 |
| "声乐老师说我挤卡，但我不知道什么时候在挤" | 无 | **市场空白** |

### 2.3 市场机会

**核心洞察**：现有竞品全部基于 F0 音高追踪（YIN/CREPE），停留在"音准评测"层面。没有任何消费级产品能实时分析发声*质量*（声带闭合、共鸣位置、发声病灶）。这是一个被学术界研究了几十年但从未被产品化的领域。

Vibesing 要做的，是把声乐教室里声乐老师的"耳朵"，变成手机里的 AI。

## 3. 目标用户

### 3.1 核心用户画像

| Persona | 年龄 | 特征 | 核心需求 | 付费意愿 |
|---------|------|------|---------|---------|
| **KTV 爱好者小明** | 18-30 | 每周唱 1-2 次，有心仪的难度歌曲但唱不上去 | 想知道"怎么才能唱上那个高音" | 中（¥20-40/月） |
| **声乐学生小红** | 15-25 | 正在上声乐课，但每周只有 1 节课 | 课后练习时想要实时反馈 | 高（¥50+/月，或老师推荐学校团购） |
| **翻唱创作者阿杰** | 20-35 | 在抖音/B 站发翻唱作品 | 提升作品质量，突破音域限制 | 高（¥50+/月） |
| **声乐老师张老师** | 30-50 | 线下/线上教学 | 给学生布置可量化的课后练习 | B 端合作 |

### 3.2 用户分层

```
Tier 1（MVP 目标）：KTV 爱好者 + 声乐初学者
  → 最大用户基数，对"诊断"功能有强新鲜感
  → 留存取决于"反馈是否有感"

Tier 2（V1.0 目标）：声乐学生 + 翻唱创作者
  → 付费意愿最强，使用频率最高
  → 需要更精细的维度（混声、共鸣、颤音）

Tier 3（V2.0 目标）：声乐老师 + 机构
  → B 端商业化，批量采购
  → 需要学员管理、进度追踪功能
```

## 4. 核心产品理念

### 4.1 三大设计原则

#### 原则一：实时优于精确

> 用户宁可要一个"大致准但立刻告诉我"的反馈，也不要一个"非常准但唱完 3 秒后才出来"的报告。

- 端到端延迟 < 100ms
- 宁可有 20% 误报率，也不能有 >200ms 延迟

#### 原则二：减法优于加法

> 不要一次告诉用户 10 个问题。每次只指出 1 个最严重的病灶。

- MVP 只做"挤卡"一件事
- 通过版本迭代逐步解锁更多维度
- 用户心智负担 = 惊艳度的最大敌人

#### 原则三：正向激励优于批评

> 用户打开 App 的动力不是"被骂"，而是"感觉自己在变好"。

- 分数默认 100 分，扣分制（不是从 0 加上去）
- 连续良好发声有奖励动画
- 红灯消失时有"恢复健康"的绿色庆祝动效
- 避免使用"错误"、"失败"等负面措辞，改用"挑战"、"调整"

### 4.2 核心用户价值主张

```
Before Vibesing: "我知道高音唱不好，但不知道哪里不对。"
After Vibesing:  "哦原来是这里在挤！一放松马上就好了！"
```

这个 "Aha Moment" 是产品成败的关键。如果用户第一次使用时没有在 30 秒内体验到这个"原来如此"的瞬间，就会流失。

## 8. 用户旅程

### 8.1 新用户首次体验（V0.1 FTUE）

```
Step 1: 下载 App，打开
  → 看到大标题"高音觉醒"和一个灰色圆形灯
  → 心理状态："这是什么？"

Step 2: 点击"开始演唱"
  → 弹出麦克风权限请求
  → 授权

Step 3: 灰灯变绿（等待演唱状态）
  → 用户开始唱一首歌的副歌
  → 绿灯亮起，分数显示 85-95

Step 4: 唱到高音段，不自觉地挤了
  → 红灯瞬间亮起！伴随轻微振动！
  → 分数骤降至 30-40
  → 用户："卧槽！？"  ← 这就是 Aha Moment

Step 5: 用户试着放松喉咙
  → 红灯消失，绿灯恢复
  → 分数回升至 80+
  → 反馈弹窗："刚才的检测准吗？"
  → 用户点"准" 👍

Step 6: 用户反复尝试：挤→红灯，放松→绿灯
  → 建立因果认知："原来这就是挤卡！"
  → 产品价值成立 ✅
```

### 8.2 成熟用户日常使用（V2.0）

```
打开 App → 查看今日训练建议（"建议先做 5 分钟唇颤音热身"）
  → 进入唇颤音练习模式
  → 跟着 App 指令做：从 C4 滑到 G4
  → 实时反馈：稳定性 ✅、音准 ✅、气息 ⚠️（末尾漏气）
  → 练习 10 分钟
  → 查看今日得分 + 对比昨天
  → "你的混声稳定度比上周提高了 12%！" → 成就感 ✅
  → 分享到朋友圈（可选）
```

## 10. 核心交互规则

### 10.1 红灯触发规则

| 规则 | 值 | 理由 |
|------|---|------|
| 连续 N 帧确认才亮灯 | 6 帧 (96ms) | 防止单帧噪音导致闪烁 |
| 连续 M 帧确认才灭灯 | 10 帧 (160ms) | 防止红灯快速闪烁式灭→亮 |
| 触发概率阈值 | 0.55 | 宁可略微敏感（用户觉得"有反应"） |
| 消除概率阈值 | 0.30 | 大滞后区间，灯亮后不容易立刻灭掉 |

### 10.2 分数显示规则

- 分数在 UI 上以整数显示（不显示小数）
- 分数变化有阻尼动画（0.2s ease-in-out），不允许瞬间跳变
- 分数颜色：≥80 绿色，50-79 黄色，<50 红色
- 静默时分数保持上一个值，不归零

### 10.3 反馈弹窗规则

- 触发时机：红灯消失（挤卡结束）后 0.5 秒
- 自动消失：5 秒后自动隐藏
- 不在以下情况弹出：
  - 红灯持续时间 < 0.5 秒（可能是噪音）
  - 上一次反馈弹窗结束不到 10 秒
  - 用户正在操作其他 UI 元素

### 10.4 动效规范

| 元素 | 动效 | 时长 |
|------|------|------|
| 红绿灯切换 | 颜色渐变 + 图标切换 | 0.3s ease-in-out |
| 红灯触发 | 外圈脉冲 ×3 + 轻振动 | 0.5s |
| 分数变化 | 数字滚动 | 0.2s |
| 能量条 | 线性插值 | 0.05s (50ms, 与帧率同步) |
| 反馈弹窗弹出 | 从中心放大 + 淡入 | 0.3s |
| 反馈弹窗消失 | 缩小 + 淡出 | 0.2s |

## 12. 数据与隐私

### 12.1 数据收集范围

| 数据 | 收集时机 | 存储位置 | 用户可控 |
|------|---------|---------|---------|
| 音频原始流 | 实时处理，不存储 | 仅在内存中 | — |
| 挤卡检测特征值（HNR/F0/Energy） | 红灯触发时 | 本地 SQLite | 可删除 |
| 用户反馈（准/不准） | 用户主动点击时 | 本地 SQLite | 可删除 |
| 2 秒音频片段（仅 V0.2+） | 红灯触发时 | 本地缓存 | 可关闭 |

### 12.2 隐私原则

1. **V0.1 不上传任何数据**：所有数据留在用户设备。
2. **未来数据上传需双重授权**：系统弹窗 + App 内开关。
3. **音频数据如需上传，必须脱敏**：仅上传特征值，不上传原始音频（除非用户明确同意）。
4. **符合 GDPR + 中国个人信息保护法（PIPL）**。

### 12.3 App Store 隐私标签

| 标签 | 值 |
|------|---|
| Data Linked to You | None (V0.1) |
| Data Used to Track You | None |
| Data Not Linked to You | 使用数据（分析）— V0.2+ 如启用上传 |

**核心竞争壁垒**：
1. **技术壁垒**：端侧实时 HNR/CPP 检测 + 多维 AI 诊断，竞品需要 6+ 个月追赶
2. **数据壁垒**：先发积累的用户反馈数据 → 更准的模型 → 更多用户（飞轮效应）
3. **品类壁垒**：第一个做"发声质量诊断"品类定义的产品 = 用户心智品牌占位

## 16. 词汇表

| 术语 | 英文 | 定义 |
|------|------|------|
| 挤卡 | Vocal Squeeze / Constriction | 唱高音时喉部肌肉过度紧张，声带过度内收，声音变"卡"变"憋" |
| 混声 | Mix Voice | 胸声和头声共振的混合状态，是唱高音的"正确方式" |
| 胸声 | Chest Voice | 低音区为主的发声类型，声带全长振动 |
| 头声 | Head Voice | 高音区为主的发声类型，声带边缘振动 |
| 假声 | Falsetto | 声带极薄振动模式，气息感重，音量小 |
| 大白嗓 | Yelling / Belting (untrained) | 不经过任何共鸣技巧直接喊叫，高频泛音缺失 |
| 喉位高 | High Larynx | 喉头上提导致声道缩短，声音变尖变紧 |
| 漏气 | Breathiness / Glottal Insufficiency | 声带闭合不完全，气息从缝隙泄出 |
| HNR | Harmonics-to-Noise Ratio | 谐噪比，衡量声音中谐波与噪声的比值（dB）。越高越"干净" |
| CPP | Cepstral Peak Prominence | 倒谱峰值突变，衡量声音周期性强度。挤卡时骤降 |
| F0 | Fundamental Frequency | 基频，即音高（Hz） |
| EMA | Exponential Moving Average | 指数移动平均，用于平滑实时信号抖动 |
| CoreML | Apple Core ML | 苹果端侧机器学习推理框架 |
| ANE | Apple Neural Engine | 苹果芯片上的神经网络专用处理器 |
| PMF | Product-Market Fit | 产品市场契合度 |
| TestFlight | — | 苹果官方的 beta 测试分发平台 |
